# Информация о репликации и шардировании

## 1) Репликация

Для БД социальной сети используется схема **master-slave** (или **primary-replica**):

- **1 master (primary)** — все операции записи (`INSERT/UPDATE/DELETE`).
- **2 slave (replica/read-only)** — операции чтения (лента, профиль, просмотры, комментарии, счётчики).

**Replication factor: 3 (1 master, 2 slave).**

---

## 2) Шардирование

Шардирование выполняется **горизонтально** и используется подход **key-based sharding by id**.

### Ключи шардирования (by id)

1. **Пользовательские данные** (`users`, `posts`, `media`, `comments`, `post_likes`, `post_views`, `subscriptions`, `friendships`, `relationships`, `user_interests`)  
   Шард-ключ: `id_user` (или связанный пользовательский id, например `id_follower`, `id_sender`).

2. **Чаты и сообщения** (`chats`, `chat_members`, `messages`)  
   Шард-ключ: `id_chat`.

### Стратегия распределения

- Используется **consistent hashing** по shard key:
  - `hash(id_user) % N` для пользовательского домена;
  - `hash(id_chat) % M` для чатов/сообщений.
- При добавлении новых шардов переносится только часть данных.

### Кросс-шардовые операции

- Избегать тяжёлых `JOIN` между разными шардами.
- Для агрегатов (рейтинги/тренды) использовать отдельный слой агрегации (ETL).
- В API и сервисах использовать роутинг на нужный шард по ключу (`id_user` / `id_chat`).

---

## 3) Итоговая схема

- Внутри каждого шарда: `mater + slave`.
- Между шардами: горизонтальное разделение по `id_user` и `id_chat`.
- Масштабирование достигается и по чтению (реплики), и по объёму/записи (шарды).
